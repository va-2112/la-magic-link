Configuration Steps
To grant Mercer a maximum of flexibility to adapt the self-registration process to each customers’ expectation, it has been agreed (March 13, 2026) that LumApps is not providing an out-of-the-box feature, but the documentation of the flow and APIs calls that need to be implemented by Mercer during the development of a LumApps extension. Once developed, Mercer will have to embed this extension, which will contain a Self-Registration form, into a public page of a public site of the platform.
Feature toggle
Prerequisites: Feature Enablement
Activate Feature Flag: LumApps Support or the organization's Technical Account Manager must be contacted to ensure the public-self-registration feature flag is enabled for the specific Organization ID.
User Pre-Provisioning Rules
Users must be pre-provisioned following specific guidelines before they can self-register.
Pre-provisioning Requirement: The endpoint is explicitly built for "pre-provisioned users with no email". This means users must have already been provisioned in LumApps with their core details: Last Name, Employee ID, and Date of Birth (formatted strictly as YYYY-MM-DD).
Note: SSN and First Name are optional.
No Existing Email: The target user's profile must not have an email address populated.
Data Accuracy: The data sent during the onboarding request must perfectly match the user's profile. While validation for First Name and Last Name is case-insensitive, all other fields require a strict, exact match.
User Directory Preparation
The system requires specific data types to map the validation rules correctly.
Create Custom Fields: Navigate to the LumApps’ user directory configuration and create a custom profile field for the user's Date of Birth, and optionally, one for their Social Security Number (SSN).
Note: Core attributes like First Name, Last Name, and Employee ID are natively built into the LumApps database, meaning the backend automatically knows where to validate them. Date of Birth and SSN are not native, which is why they must be explicitly created as custom fields.
Set Field Type: The Date of Birth field type must be explicitly set to date.
Obtain Field IDs: To find the system-generated UUIDs for the new fields, make a GET request to the /v2/organizations/{organizationId}/userprofile/field-definitions endpoint.
Authorization: Requires a Bearer token with all.read or all.admin privileges.
Locate the new Date of Birth (and SSN) fields in the JSON response array and copy their respective id values.
API Configuration Setup (Field Mapping)
Because every organization creates custom fields differently, the validation engine needs to be told exactly which unique UUID corresponds to the Date of Birth and SSN. (Native fields do not require this mapping).
Endpoint: PUT /v2/organizations/{organizationId}/self-onboardings/configuration.
Authorization: Requires a Bearer token with all.admin privileges.
Payload Structure: Required: Provide the dateOfBirthId (the UUID retrieved in/userprofile/field-definitions endpoint) and specify any additionalValidationFields. (Send an empty array [] if only validating the core fields).
Optional (SSN Configuration): If SSN is used as an additional security checkpoint, both the mapping and the rule must be provided:
The Map (ssnId): Provide the UUID retrieved in the previous step. This simply tells the database where the SSN is stored.
The Rule (additionalValidationFields): The string "ssn" must be explicitly added to this array. This acts as the strict security checklist, instructing the validation engine that an SSN match is mandatory before sending a magic link. Sending the ssnId alone gives the system the location of the data, but no instruction to actually enforce it.

Magic Link Generation (Execution)
Execute the non-authenticated request to validate the user and send the mobile OTP (One-Time Password).
Endpoint: POST /v2/organizations/{organizationId}/self-onboardings/magic-links.
Authorization: No Auth
Payload Fields: Required Fields: lastName, dateOfBirth (formatted strictly as YYYY-MM-DD), employeeId, and email (which acts as the destination for the OTP link).
Optional Fields: firstName and ssn (these can be included if the configuration was set up to validate them).
Confirmation: A successful request matching the above data will return a 204 No Content and dispatch the magic link.
